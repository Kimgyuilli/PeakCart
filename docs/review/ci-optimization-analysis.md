# CI 파이프라인 개선 가능성 분석 리포트

> 작성일: 2026-04-14
> 대상: `.github/workflows/ci.yml`
> 기준: PR #16 run [24390371115](https://github.com/Kimgyuilli/PeakCart/actions/runs/24390371115), 현재 테스트 구조, 통합 테스트 실행 방식
> 목적: 이 작업이 지금 수행할 가치가 있는지 판단하고, 가치가 있다면 어떤 순서로 안전하게 진행할지 결정

---

## 1. 결론

### 판단

**이 작업은 할 가치가 있다.**

다만 성격은 "즉시 필수 최적화"가 아니라, **Phase 3 마무리 단계에서 수행할 테스트 인프라 정비 작업**에 가깝다.

### 이유

현재 CI에서 가장 무거운 구간은 테스트이고, 특히 통합 테스트가 클래스별로 Testcontainers 를 중복 기동하는 구조다. 이 문제는 단순히 몇 초를 줄이는 수준이 아니라 아래 구조 문제를 함께 갖고 있다.

- 통합 테스트마다 MySQL / Redis / Kafka 컨테이너 선언이 반복됨
- 공통 베이스 클래스와 실행 규약이 없어 신규 테스트 작성 방식이 분산됨
- 일부 테스트는 개별 cleanup, 일부는 fresh container 격리에 기대고 있어 테스트 인프라 정책이 일관되지 않음
- 추후 통합 테스트가 늘어날수록 CI 시간과 유지보수 비용이 함께 증가할 가능성이 높음

즉 이 작업의 가치는 단순 속도 개선보다도 **테스트 인프라 표준화**에 있다.

### 지금 당장 해야 하는가?

**반드시 지금은 아니다.**

현재 CI 6분대는 불편하지만, 진행 중인 기능 작업을 멈추고 바로 착수해야 할 정도의 블로커는 아니다. 따라서 기능 개발 흐름이 중요한 시점이라면 뒤로 미루는 것이 맞고, Phase 3 마무리 시점에 "테스트 인프라 정비" 작업으로 묶어 처리하는 것이 가장 합리적이다.

---

## 2. 현재 상태 요약

현재 CI는 단일 `build` job 에서 `./gradlew build --no-daemon` 을 실행한다.

- 단위 테스트와 통합 테스트가 하나의 `test` 태스크 안에서 함께 돈다
- 통합 테스트 7개가 클래스별로 `@Testcontainers` + `static @Container` 를 사용한다
- 공통 `AbstractIntegrationTest` 베이스 클래스가 없다
- 일부 테스트는 `@TestPropertySource`, `@Import(TestConfig.class)`, 개별 SQL cleanup 을 사용한다

이 구조는 다음 의미를 갖는다.

- 지금도 테스트는 동작하지만, 실행 규약이 흩어져 있다
- 통합 테스트 수가 늘수록 속도와 복잡도가 함께 악화될 가능성이 높다
- CI 최적화의 핵심은 Gradle 캐시보다 **통합 테스트 구조 정리**에 있다

---

## 3. 가치 판단

### 가치가 있는 이유

이 작업은 다음 네 가지 측면에서 가치가 있다.

1. **CI 피드백 개선**
   - 테스트 인프라 중복 비용을 줄일 여지가 있다
   - 향후 통합 테스트 증가에 따른 시간 악화를 완화할 수 있다

2. **테스트 구조 표준화**
   - 통합 테스트의 컨테이너 선언, cleanup, 공통 설정 방식을 하나의 규약으로 정리할 수 있다
   - 신규 테스트 작성 시 "어떻게 시작해야 하는가"가 명확해진다

3. **유지보수 비용 절감**
   - 지금은 각 테스트가 자체적으로 인프라를 들고 있어 수정 포인트가 분산돼 있다
   - 베이스 클래스 도입 시 공통 변경을 한 곳에 집중시킬 수 있다

4. **추후 개선의 기반 확보**
   - 이후 unit/integration 분리, fail-fast 강화, 병렬 전략 검토 등 후속 작업이 쉬워진다

### 가치가 제한되는 이유

반대로 이 작업이 지금 최우선이 아닌 이유도 분명하다.

- 현재 CI 시간이 아주 길지는 않다
- 기능 개발의 직접 산출물을 만들지는 않는다
- 잘못 설계하면 flaky test 를 늘릴 수 있다
- Kafka, DB, Redis 상태 격리 설계를 대충 하면 오히려 신뢰성이 나빠진다

즉 **가치는 높지만, 우선순위는 상황 의존적**이다.

## 4. 권고 우선순위

### 권고

**권장 순위: "지금 바로"가 아니라 "Phase 3 마무리 직후 또는 직전의 정비 작업"**

### 지금 하는 것이 맞는 경우

- 최근 CI 대기 시간이 실제로 개발 리듬을 자주 끊고 있을 때
- 통합 테스트가 추가될 예정이라 지금 구조를 더 키우는 것이 부담일 때
- 테스트 인프라 규약 정리가 다음 작업들에도 도움이 될 때

### 지금 하지 않는 것이 맞는 경우

- 현재 가장 중요한 것이 기능 완성과 검증 범위 확대일 때
- CI 6분대가 아직 감내 가능한 수준일 때
- 테스트 인프라 정비보다 서비스 기능/운영 과제가 더 시급할 때

### 최종 판단

**"언젠가 하면 좋은 일" 수준은 아니다.**
구조상 분명히 손볼 가치가 있는 작업이다.

다만 **"지금 기능 작업을 멈추고 바로 들어가야 하는 일"도 아니다.**
현재 프로젝트 맥락에서는 **Phase 3 마무리 단계에 묶어 처리하는 것이 가장 적절하다.**

## 5. 어떤 방식으로 해야 하는가

이 작업은 한 번에 크게 하지 말고, 목표를 두 단계로 나눠야 한다.

### 1차 목표

**통합 테스트 인프라 공통화**

여기서의 목적은 아래다.

- 컨테이너 선언 중복 제거
- 통합 테스트 베이스 클래스 도입
- cleanup 정책을 공통 규약으로 정리
- 신규 테스트 작성 규칙 문서화

즉 1차 목표는 **"테스트 인프라를 표준화한다"** 이다.

### 2차 목표

**추가 최적화**

1차가 안정화된 뒤에 아래를 검토한다.

- Spring context 재사용 확대
- unit / integration job 분리
- fail-fast 개선
- 단위 테스트 전용 병렬화

즉 2차 목표는 **"표준화된 구조 위에서 속도를 더 다듬는다"** 이다.

### 중요한 원칙

**컨테이너 재사용과 컨텍스트 재사용을 한 번에 달성하려 하지 말 것.**

현재 테스트들은 `@Import`, `@TestPropertySource`, 웹 환경 차이, 개별 SQL cleanup 등으로 서로 실행 조건이 다르다. 따라서 처음부터 모든 테스트를 하나의 완전 공유 구조로 밀어 넣으면 속도보다 flaky 위험이 더 커질 수 있다.

처음에는 **컨테이너 중복 제거와 공통 실행 규약 정리**에만 집중하는 것이 맞다.

## 6. 작업 계획

### Step 1. 사전 확인

착수 전에 아래를 확인한다.

- 통합 테스트 7개 목록과 각 테스트의 인프라 요구사항 정리
- 각 테스트의 cleanup 방식 정리
- `@Import`, `@TestPropertySource`, 웹 환경 차이 정리
- `DlqIntegrationTest`, `OutboxKafkaIntegrationTest`, `IdempotencyIntegrationTest` 처럼 Kafka 의존도가 높은 테스트를 별도 표시

목적은 "어떤 테스트가 쉽게 옮겨지고, 어떤 테스트가 까다로운지" 미리 구분하는 것이다.

### Step 2. 공통 베이스 설계

`AbstractIntegrationTest` 또는 이에 준하는 공통 베이스를 설계한다.

포함 대상:

- MySQL / Redis / Kafka 공통 컨테이너 선언
- 공통 property 연결 방식
- 테스트 전 상태 초기화 규약
- 필요 시 공통 helper

이 단계에서 결정할 핵심은 하나다.

**상태 격리를 무엇으로 보장할 것인가**

- DB truncate
- Redis flush
- Kafka topic/group 관리

이 기준이 모호하면 이후 단계에서 반드시 흔들린다.

### Step 3. 쉬운 테스트부터 순차 마이그레이션

가장 단순한 테스트부터 옮긴다.

권장 순서 예시:

1. `ProductCacheIntegrationTest`
2. `InventoryConcurrencyTest`
3. `ShedLockIntegrationTest`
4. `ObservabilityMetricsIntegrationTest`
5. `OutboxKafkaIntegrationTest`
6. `IdempotencyIntegrationTest`
7. `DlqIntegrationTest`

이 순서의 이유는 Kafka 시나리오가 강한 테스트를 뒤로 미뤄서 초기 리스크를 줄이기 위함이다.

### Step 4. 안정화 검증

마이그레이션 후에는 속도보다 먼저 안정성을 확인한다.

- 로컬 반복 실행
- CI 3회 이상 반복 확인
- 테스트 간 상태 누수 여부 확인
- 간헐 실패 여부 확인

이 단계에서 flaky 가 보이면 속도 개선보다 안정화가 우선이다.

### Step 5. 선택적 후속 최적화

1차 정비가 끝난 뒤 필요할 때만 아래를 붙인다.

- unit / integration job 분리
- fail-fast 개선
- 단위 테스트 전용 `maxParallelForks`
- Spring context 재사용 범위 확대

## 7. 리스크

### 가장 큰 리스크

**테스트 상태 누수**

특히 위험한 영역은 Kafka 다.

- 이전 테스트 메시지가 다음 테스트에 영향
- consumer group / offset 잔존
- DLQ 검증 테스트의 고정 토픽 의존

DB 와 Redis 도 위험하지만, 상대적으로 cleanup 전략이 더 단순하다.

### 두 번째 리스크

**규약은 만들었는데 실제로는 일관되게 쓰이지 않는 상태**

예를 들어 일부 테스트는 베이스 클래스를 쓰고, 일부는 기존 방식과 수동 SQL cleanup 을 유지하면 구조가 오히려 더 복잡해진다.

따라서 도입 후에는 반드시 다음 둘 중 하나여야 한다.

- 새 규약으로 통일
- 예외 테스트를 명시적으로 문서화

### 세 번째 리스크

**속도 개선에 집착하다가 테스트 신뢰성을 해치는 것**

이 작업의 성공 기준은 단순히 "더 빨라졌다"가 아니다.

---

## 8. 성공 기준

이 작업이 성공했다고 보려면 아래 조건을 만족해야 한다.

1. 통합 테스트 공통 실행 규약이 문서와 코드에 모두 존재한다
2. 신규 통합 테스트 작성 방식이 명확해진다
3. CI 또는 로컬 테스트 시간이 체감 개선된다
4. flaky test 가 증가하지 않는다
5. Kafka/DB/Redis 상태 격리 방식이 명시적으로 관리된다

속도 개선만 있고 flaky 가 늘면 실패다.
공통 베이스만 만들고 예외가 계속 생기면 그것도 실패다.

## 9. 최종 권고

현재 판단은 아래와 같다.

- **이 작업은 수행할 가치가 있다**
- **핵심 가치는 속도 그 자체보다 테스트 인프라 표준화에 있다**
- **지금 즉시 최우선으로 착수할 정도는 아니다**
- **Phase 3 마무리 단계에서 별도 정비 작업으로 수행하는 것이 가장 적절하다**
- **착수 시 1차 목표는 통합 테스트 인프라 공통화, 2차 목표는 추가 최적화로 분리해야 한다**

즉, 이 작업은 "할까 말까"의 문제가 아니라 **"지금 할까, 조금 뒤에 구조적으로 할까"의 문제**다.

현재 프로젝트 맥락에서는 **조금 뒤에 구조적으로 하는 쪽이 맞다.**
